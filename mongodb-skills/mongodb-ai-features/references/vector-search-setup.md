# Atlas Vector Search Setup Guide

Step-by-step guide to setting up MongoDB Atlas Vector Search for the RAG pipeline. Covers cluster requirements, index creation, testing, and common errors.

## Prerequisites

- MongoDB Atlas cluster (M0 free tier or higher)
- Atlas account with Project Owner or Organization Owner role
- Documents with embedding vectors stored in a collection

## Step 1: Create an Atlas Cluster

If you do not already have a cluster, create one in the Atlas UI or via the Admin API. Vector Search is available on all cluster tiers including M0 (free tier).

Key requirements:
- MongoDB version 6.0.11+ or 7.0+ (for `$vectorSearch` aggregation stage)
- The cluster must be in a supported cloud region

## Step 2: Prepare Your Collection

The RAG pipeline stores documents in the `ragdocuments` collection with this schema:

```javascript
{
  content: "The chunk of text content...",
  contentHash: "sha256-hash-of-content",
  accessLevel: "public",           // or "authenticated"
  source: {
    filePath: "docs/getting-started.md",
    title: "Getting Started Guide",
    section: "Installation",
    category: "getting-started",
    url: "/docs/getting-started",
    type: "docs"
  },
  chunk: {
    index: 0,
    totalChunks: 5,
    tokens: 128
  },
  embedding: [0.0123, -0.0456, ...],   // 1024-dimensional float array
  ingestion: {
    runId: "run-uuid",
    ingestedAt: ISODate("2025-01-15T10:30:00Z"),
    ingestedBy: ObjectId("..."),
    version: 1
  }
}
```

The `embedding` field is a 1024-dimensional array of floats generated by Voyage AI's `voyage-4-large` model.

## Step 3: Create the Vector Search Index

### Option A: Via Atlas UI

1. Navigate to your cluster in the Atlas UI
2. Click the **"Atlas Search"** tab (or **"Search"** in newer UI versions)
3. Click **"Create Search Index"**
4. Select **"Atlas Vector Search"** (JSON Editor)
5. Select the `ragdocuments` collection
6. Paste the index definition below
7. Click **"Create Index"**

### Option B: Via Atlas Admin API

```bash
curl -X POST \
  "https://cloud.mongodb.com/api/atlas/v2/groups/{groupId}/clusters/{clusterName}/search/indexes" \
  --digest -u "publicKey:privateKey" \
  -H "Accept: application/vnd.atlas.2025-03-12+json" \
  -H "Content-Type: application/json" \
  -d @vector-search-index.json
```

### Option C: Via mongosh

```javascript
db.ragdocuments.createSearchIndex(
  "rag_document_vector",
  "vectorSearch",
  {
    fields: [
      {
        type: "vector",
        path: "embedding",
        numDimensions: 1024,
        similarity: "cosine"
      },
      {
        type: "filter",
        path: "accessLevel"
      },
      {
        type: "filter",
        path: "source.category"
      }
    ]
  }
);
```

### Index Definition JSON

```json
{
  "name": "rag_document_vector",
  "type": "vectorSearch",
  "definition": {
    "fields": [
      {
        "type": "vector",
        "path": "embedding",
        "numDimensions": 1024,
        "similarity": "cosine"
      },
      {
        "type": "filter",
        "path": "accessLevel"
      },
      {
        "type": "filter",
        "path": "source.category"
      }
    ]
  }
}
```

**Field explanations:**

| Field                | Purpose                                                    |
|----------------------|------------------------------------------------------------|
| `type: "vector"`     | Declares this field contains vector embeddings             |
| `path: "embedding"`  | The document field that holds the embedding array          |
| `numDimensions: 1024`| Must match the embedding model's output dimensions         |
| `similarity: "cosine"` | Distance metric for similarity comparison               |
| `type: "filter"`     | Enables pre-filtering on this field during vector search   |

**Similarity metrics:**
- `cosine`: Best for normalized embeddings (Voyage AI normalizes by default). Measures angle between vectors.
- `euclidean`: Measures straight-line distance. Better for embeddings where magnitude matters.
- `dotProduct`: Fastest computation. Equivalent to cosine for normalized vectors.

Use `cosine` unless you have a specific reason for another metric.

## Step 4: Wait for Index Build

After creating the index, Atlas builds it in the background. Monitor the status:

- **Atlas UI**: The index status shows as "Building" then changes to "Active"
- **API**: Poll the search index endpoint until `status === "READY"`
- **Build time**: Depends on collection size. For 10,000 documents, expect 1-5 minutes. For 1M+ documents, expect 30-60 minutes.

The index must be in `READY`/`Active` status before `$vectorSearch` queries will return results.

## Step 5: Test with $vectorSearch

Once the index is active, test with an aggregation pipeline:

```javascript
// In mongosh or your application
db.ragdocuments.aggregate([
  {
    $vectorSearch: {
      index: "rag_document_vector",
      path: "embedding",
      queryVector: [0.0123, -0.0456, ...],  // 1024-dim query embedding
      numCandidates: 150,
      limit: 5
    }
  },
  {
    $project: {
      content: 1,
      source: 1,
      score: { $meta: "vectorSearchScore" }
    }
  }
]);
```

**Parameters:**

| Parameter       | Description                                              |
|-----------------|----------------------------------------------------------|
| `index`         | Name of the vector search index                          |
| `path`          | Field containing the embedding vectors                   |
| `queryVector`   | The query embedding (must match numDimensions)           |
| `numCandidates` | Number of candidates to consider (higher = more accurate, slower) |
| `limit`         | Maximum results to return                                |
| `filter`        | Optional pre-filter (uses filter fields defined in index) |

### With Pre-Filtering

```javascript
db.ragdocuments.aggregate([
  {
    $vectorSearch: {
      index: "rag_document_vector",
      path: "embedding",
      queryVector: queryEmbedding,
      numCandidates: 150,
      limit: 15,
      filter: {
        accessLevel: "public"
      }
    }
  },
  {
    $project: {
      content: 1,
      source: 1,
      score: { $meta: "vectorSearchScore" }
    }
  }
]);
```

Pre-filtering applies BEFORE the vector search, reducing the candidate set. Only fields declared as `type: "filter"` in the index definition can be used.

## Step 6: Integration with the Platform

The retrieval service (`src/lib/rag/retrieval.ts`) wraps the aggregation pipeline:

```typescript
const pipeline = [
  {
    $vectorSearch: {
      index: "rag_document_vector",
      path: "embedding",
      queryVector: queryEmbedding,
      numCandidates: topK * 30,      // Over-fetch for category boosting
      limit: topK * 3,
      ...(Object.keys(filter).length > 0 ? { filter } : {}),
    },
  },
  {
    $project: {
      content: 1, source: 1,
      score: { $meta: "vectorSearchScore" }
    }
  },
];

const results = await RagDocumentModel.aggregate(pipeline);
```

After vector search, results are post-processed with category boosting (e.g., "events" category gets a 1.6x score multiplier) and filtered by a minimum score threshold (default 0.7).

## Common Errors and Fixes

### Error: Zero results returned

**Cause:** The vector search index does not exist or is still building.

**Fix:** Check the index status in Atlas UI. Wait for it to reach "Active" status. Verify the index name matches exactly (`rag_document_vector`).

### Error: "queryVector must have X dimensions"

**Cause:** The query embedding dimensions do not match `numDimensions` in the index definition.

**Fix:** Ensure you are using the same embedding model (voyage-4-large = 1024 dims) for both documents and queries. Check that the index specifies `numDimensions: 1024`.

### Error: "field X is not indexed for filtering"

**Cause:** You are trying to filter on a field that was not declared as `type: "filter"` in the index definition.

**Fix:** Add the field to the index definition as a filter field and rebuild the index.

### Error: "$vectorSearch is not allowed"

**Cause:** The MongoDB version does not support `$vectorSearch`, or Atlas Search is not available on this cluster tier.

**Fix:** Ensure MongoDB version is 6.0.11+ or 7.0+. `$vectorSearch` is an Atlas-specific aggregation stage -- it does not work on self-hosted MongoDB.

### Error: Slow queries (>2 seconds)

**Cause:** `numCandidates` is too high, or the collection is very large without adequate index optimization.

**Fix:** Reduce `numCandidates` (start with `limit * 20`). Use pre-filters to reduce the search space. Consider upgrading the cluster tier for larger collections.

### Error: Poor retrieval quality

**Cause:** Mixing document/query input types, wrong chunking strategy, or stale embeddings.

**Fix:**
1. Verify you use `input_type: "document"` for ingestion and `input_type: "query"` for search
2. Check chunk sizes (300-500 tokens is typically optimal)
3. Re-index if the embedding model changed
4. Increase `numCandidates` for better recall
5. Verify the content hash check is not skipping re-embedding of updated documents
