# Workshop Design Patterns

Proven structures and pacing from 50+ MongoDB developer days.

## The Golden Ratio: 40-50-10

- **40%** lecture/demonstration
- **50%** hands-on exercises
- **10%** breaks

Adults learn by doing, not watching. Maximize hands-on time.

## Pattern 1: Hook-Demo-Practice-Review

**Hook (5 min):** Compelling demo of what they'll build
**Demo (25 min):** Live coding with narration
**Practice (45 min):** Guided hands-on exercise
**Review (15 min):** Debrief, Q&A, common pitfalls

**Total:** 90 minutes

**Use for:** Technical skill workshops

## Pattern 2: Problem-Solution-Practice

**Problem (15 min):** Present real-world challenge
**Solution (30 min):** Show MongoDB approach
**Practice (60 min):** Participants solve similar problem

**Total:** 105 minutes

**Use for:** Architecture/design workshops

## Pattern 3: Layered Learning (for mixed audiences)

**Core (morning):** Fundamentals (everyone together)
**Breakouts (afternoon):** Beginner track vs Advanced track

**Use for:** Large mixed-level audiences

## Pacing Guidelines

### Lecture Chunks
- **10-15 min max** before interaction
- Use "you try" checkpoints every 15 minutes
- Never go >30 min without hands-on activity

### Exercise Length
- **Beginner:** 30-45 min (highly scaffolded)
- **Intermediate:** 45-60 min (moderate guidance)
- **Advanced:** 60-90 min (open-ended)

### Break Frequency
- **Every 60-90 min minimum**
- 15 min breaks
- 60 min lunch (full-day only)

## The "Hook" Opening

Start with impact, not agenda slides.

**Bad:**
> "Welcome. Here's the agenda. First topic is..."

**Good:**
> "Watch this." [10-min demo of cool result]
> "That's what you'll build today. Let's go."

## Progressive Disclosure

Don't show full solution upfront. Build up complexity:

**Exercise 1:** Basic insert/find (success builds confidence)
**Exercise 2:** Add indexes (build on Exercise 1)
**Exercise 3:** Add aggregation (build on Exercise 2)
**Exercise 4:** Add transactions (complete application)

## Scaffolding Levels

| Level | Starter Code | Guidance | Independence |
|-------|--------------|----------|--------------|
| **Beginner** | 90% | Detailed TODOs with hints | Low |
| **Intermediate** | 70% | Clear TODOs, minimal hints | Medium |
| **Advanced** | 50% | Problem statement only | High |

## Common Failure Modes

### ❌ Too Much Content
Cramming 8 topics into 6 hours. Result: Rushed, shallow, no retention.

**Fix:** 4-5 topics maximum. Go deep, not wide.

### ❌ Passive Learning
Talking for 2 hours straight.

**Fix:** Interaction every 15 minutes.

### ❌ Broken Exercises
Starter code doesn't run, dependencies missing, broken URLs.

**Fix:** Test on fresh machine 24 hours before workshop.

### ❌ No Safety Net
Participants get stuck, no way to catch up.

**Fix:** Provide working solution to load if completely stuck.

## Fast Finisher Strategy

Always have extension challenges ready:

```markdown
## Exercise Complete? Try These:
1. Add validation rules
2. Create compound indexes
3. Optimize aggregation pipeline
4. Add error handling
```

Prevents boredom and keeps advanced participants engaged.

## Time Buffer

Plan for 80% of available time. Things always take longer:
- Setup issues
- Questions
- Debugging

**Example:** 6-hour workshop = plan 5 hours of content.

## Feedback Collection

**During:**
- Read the room (confused faces? slow down)
- Informal "thumbs up if you're done" checks

**After:**
- Quick survey (5 questions max)
- "One thing to improve" question

## Follow-Up Pattern

**Same day:** Share slides + exercise repo
**Next day:** Thank you email + resources
**One week:** Check-in email
**One month:** "How's it going?" follow-up

Builds relationship beyond one-day event.
