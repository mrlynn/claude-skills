# Judging Best Practices

## Core Principles

### 1. Fair and Consistent
- Score against rubric, not other teams
- Apply same criteria to all submissions
- Recalibrate after first few submissions

### 2. Constructive and Encouraging
- Start with strengths
- Frame improvements positively
- Acknowledge constraints (time, resources)

### 3. Specific and Actionable
- Avoid vague feedback
- Give concrete suggestions
- Prioritize high-impact improvements

## DO:

✅ **Celebrate effort**
> "You built a working prototype in 24 hours - impressive!"

✅ **Be specific**
> "The MongoDB aggregation pipeline was well-designed."

✅ **Acknowledge constraints**
> "Given the time limit, prioritization was smart."

## DON'T:

❌ **Harsh criticism**
> "This code is terrible."

❌ **Vague feedback**
> "Needs improvement."

❌ **Compare to production**
> "This wouldn't pass code review."

## Tie-Breaking

1. Innovation (novel > derivative)
2. Completeness (working > concept)
3. Presentation (clear > confusing)
4. Technical quality (clean > hacky)

## Golden Rule

Your feedback might determine if a team continues their project. Make it count.
